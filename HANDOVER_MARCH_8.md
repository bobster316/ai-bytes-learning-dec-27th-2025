## Issues Found

### 🔴 Issue 1: Veo Blocking Generation (CRITICAL — blocks all lesson saving)

**Problem:** `veoVideoService.generateVideo()` in `app/api/course/generate-v2/route.ts` (lines 321–335) is called synchronously per lesson. Each call takes 1–6 minutes. With 4 lessons (2 modules × 2 lessons), the route exceeds its `maxDuration = 300` (5 min) before any lessons are saved.

**Location:** `app/api/course/generate-v2/route.ts` lines 321–335

```typescript
// CURRENT (blocking — causes timeout before lessons are saved)
const veoResult = await veoVideoService.generateVideo(v.videoPrompt, v.caption || '');
if (veoResult) v.videoUrl = veoResult.url;
```

**Fix (Option A — Recommended):** Make Veo non-blocking. Save the lesson immediately, then update the video URL asynchronously after generation completes.

```typescript
// NON-BLOCKING — save lesson first, video fills in later
veoVideoService.generateVideo(v.videoPrompt, v.caption || '').then(async (veoResult) => {
    if (veoResult?.url) {
        // Re-fetch the saved lesson's blocks, update the video block URL, re-save
        const { data: saved } = await supabase
            .from('course_lessons')
            .select('id, content_blocks')
            .eq('topic_id', topicId)
            .eq('title', lessonPlan.lessonTitle)
            .single();
        if (saved) {
            const updatedBlocks = (saved.content_blocks || []).map((b: any) =>
                b.type === 'video_snippet' && b.videoPrompt === v.videoPrompt
                    ? { ...b, videoUrl: veoResult.url }
                    : b
            );
            await supabase.from('course_lessons')
                .update({ content_blocks: updatedBlocks })
                .eq('id', saved.id);
        }
    }
}).catch(err => console.warn('[API-V2] Veo async failed:', err));
// Do NOT await — continue to save lesson immediately
```

**Fix (Option B — Quick dev bypass):** Add env flag in `.env.local`:
```
SKIP_VEO=true
```
Then wrap the Veo block:
```typescript
if (!DRY_RUN && compiledLesson.blocks && process.env.SKIP_VEO !== 'true') {
    // Veo block...
}
```

### 🟡 Issue 2: Callout Blocks — `variant` Missing (Low Priority — Renders Fine)

Blocks 9 and 12 in lesson 3387 were generated without a `variant` field:
```json
{ "type": "callout", "icon": "warning", "text": "...", "order": 9 }
```

**Actual impact:** `CalloutBox` component (`components/course/blocks/callout-box.tsx` line 32) already uses `text || body`, so the content renders. Without `variant`, `isTip = false` so it shows as amber/warning style. This is acceptable fallback behaviour — **no fix required unless visual consistency matters**.

If you want to improve it, add variant derivation from icon:
```typescript
const finalVariant = variant || (icon === 'warning' || icon === '⚠️' ? 'warning' : 'tip');
const isTip = finalVariant === 'tip';
```

### 🟠 Issue 3: Dashboard SSR Auth Redirect

**Problem:** `/dashboard` redirects to `/auth/signin` when authenticated via magic link. Magic links set session in browser localStorage. The Next.js SSR Supabase client reads from cookies. The hash-based token (`#access_token=...`) is only processed client-side by Supabase JS.

**Impact:** Dashboard inaccessible without a cookie-based session (email/password login). Magic link works fine for production users because the Supabase JS client processes the hash on the client and sets cookies — but Playwright automation and SSR are affected.

**Fix options:**
- Use email/password login for Playwright tests
- Implement a `/api/auth/session-exchange` endpoint that takes an access token and sets it as a cookie
